# Xentric AI Webhook Receiver

Production-grade multi-tenant CRM webhook receiver built with NestJS, Fastify, TypeScript, PostgreSQL 16, Knex, Pino, Vitest, Supertest, Docker Compose, and Swagger.

This project is intentionally self-contained. It treats the assignment's webhook specification as the third-party CRM contract, generates its own test data, and verifies correctness with real PostgreSQL integration tests.

## Feature Checklist

- NestJS with Fastify adapter and strict TypeScript
- HMAC-SHA256 webhook signature verification over the raw request body
- Timing-safe signature comparison with `crypto.timingSafeEqual()`
- Durable idempotency through `processed_events`
- Atomic PostgreSQL upsert so the newest contact update wins under concurrency
- Out-of-order event protection using `updated_at`
- PostgreSQL Row Level Security for tenant isolation
- Non-superuser `app_user` role for application queries
- Bearer-token protected contacts read API
- Structured JSON logging with webhook disposition tracking
- Swagger UI for manual API testing
- Docker Compose one-command startup
- Integration tests covering webhook correctness, auth, concurrency, and RLS

## Architecture

Webhook flow:

1. A generic CRM sends `POST /webhooks/crm/contact-updated` with tenant, event id, and HMAC signature headers.
2. Nest/Fastify preserves the raw request body using `rawBody: true`.
3. `WebhookService` verifies the HMAC using the configured tenant secret.
4. `WebhookRepository` opens a transaction, sets `app.current_client`, inserts the event id into `processed_events`, and atomically upserts the contact.
5. PostgreSQL enforces idempotency, ordering, concurrency safety, and tenant isolation.
6. `LoggerService` emits one structured JSON log with the webhook disposition.

Read flow:

1. Client calls `GET /contacts/:client_id` with a bearer token.
2. `AuthGuard` maps the token to a tenant.
3. `ContactsService` rejects cross-tenant URL access with `403`.
4. `ContactsRepository` sets `app.current_client` and queries `contacts` without an application-level tenant `WHERE` filter.
5. PostgreSQL RLS enforces tenant isolation at the database layer.

## Assumptions

- No real CRM sandbox is required; the assignment webhook specification is treated as the external CRM contract.
- Demo tenants are `client_a`, `client_b`, and `client_c`.
- Demo secrets and tokens are configured through Docker Compose and `.env.example`.
- Webhook `event_id` values are UUIDs generated by the sender or tests.
- `updated_at` is the source-of-truth ordering field for contact updates.
- Offset pagination is used because the assignment requires it.
- Production secrets would be managed outside source control in a real deployment.

## PowerShell Webhook Test

This signs the exact file bytes that `curl` sends, avoiding quoting or encoding mismatches:

```powershell
$bodyPath = ".\webhook-body.json"
$body = '{"contact_id":"real-contact-123","email":"real.user@example.com","phone":"+15551234567","first_name":"Real","last_name":"User","status":"active","updated_at":"2026-06-21T10:00:00.000Z"}'

[System.IO.File]::WriteAllText($bodyPath, $body, [System.Text.UTF8Encoding]::new($false))

$sig = node -e "const fs=require('fs'); const crypto=require('crypto'); const file=process.argv[1]; const body=fs.readFileSync(file); console.log(crypto.createHmac('sha256','client-a-secret').update(body).digest('hex'))" $bodyPath

curl.exe -i -X POST "http://localhost:3000/webhooks/crm/contact-updated" `
  -H "Content-Type: application/json" `
  -H "X-CRM-Client-Id: client_a" `
  -H "X-CRM-Event-Id: 55555555-5555-4555-8555-555555555555" `
  -H "X-CRM-Signature: $sig" `
  --data-binary "@$bodyPath"
```

Then verify the contact:

```powershell
curl.exe -i -H "Authorization: Bearer client-a-token" "http://localhost:3000/contacts/client_a?page=1&limit=50"
```

## Design Decisions

### Why Fastify

The assignment mandates NestJS with the Fastify adapter. The application still follows standard NestJS patterns: modules, controllers, services, guards, DTOs, providers, and dependency injection.

### Why Knex

Knex provides explicit SQL control without using a banned ORM. That is important here because the core behavior depends on PostgreSQL features such as composite primary keys, `ON CONFLICT`, transaction-local settings, least-privilege roles, and RLS policies.

### Why PostgreSQL RLS

Tenant isolation belongs at the database boundary. The read repository intentionally does not add an application-level `WHERE client_id = ...` filter; PostgreSQL enforces row visibility through RLS.

Before tenant-scoped reads or writes, the app executes:

```sql
select set_config('app.current_client', '<client_id>', true);
```

The `true` flag makes the setting transaction-local. RLS policies compare each contact row against:

```sql
current_setting('app.current_client', true)
```

### Why `processed_events`

A database table is durable across process restarts and horizontal scaling. In-memory deduplication would lose state and allow duplicate mutations after restarts.

## Database Design

`contacts` uses `(client_id, contact_id)` as a composite primary key because contact ids are tenant-scoped. It stores source contact fields, the source `updated_at`, and operational timestamps.

`processed_events` stores each webhook event id once for durable idempotency.

Indexes:

- `contacts(client_id)` supports tenant-oriented access and RLS checks.
- `contacts(updated_at)` supports newest-first ordering.
- `processed_events(client_id)` supports tenant-level audit and inspection.

## Idempotency And Concurrency

Webhook processing runs in one transaction:

1. Insert `event_id` into `processed_events` with `ON CONFLICT DO NOTHING`.
2. If no row was inserted, return success and skip contact mutation.
3. If inserted, upsert the contact.

The contact upsert is atomic:

```sql
INSERT INTO contacts (...)
VALUES (...)
ON CONFLICT (client_id, contact_id)
DO UPDATE SET ...
WHERE excluded.updated_at > contacts.updated_at;
```

This makes concurrent delivery deterministic: the newest `updated_at` wins, and older events are ignored.

## Security Considerations

- HMAC-SHA256 is calculated over the raw request body.
- Signature comparison is timing-safe.
- Unknown clients or invalid signatures return `401`.
- Bearer tokens map to tenants and cross-tenant reads return `403`.
- PostgreSQL RLS protects tenant data even if a query forgets tenant filtering.
- The application connects as non-superuser `app_user`.
- DTO validation rejects malformed payloads and unknown body fields.
- Demo secrets are for local execution only and should be replaced in production.

## Structured Logging

Every webhook emits exactly one structured Pino log with one disposition:

- `processed`
- `duplicate_ignored`
- `out_of_order_ignored`
- `signature_invalid`
- `error`

Log fields include `event_id`, `client_id`, `contact_id`, `disposition`, and an ISO timestamp.

## Running Without Docker

Start PostgreSQL 16 locally with database `webhooks`, then configure:

```bash
DATABASE_URL=postgres://app_user:app_password@localhost:5432/webhooks
MIGRATION_DATABASE_URL=postgres://postgres:postgres@localhost:5432/webhooks
APP_DB_USER=app_user
APP_DB_PASSWORD=app_password
CLIENT_A_SECRET=client-a-secret
CLIENT_B_SECRET=client-b-secret
CLIENT_A_TOKEN=client-a-token
CLIENT_B_TOKEN=client-b-token
```

Then run:

```bash
npm install
npm run migrate
npm run start:dev
```

## Open Questions Before Production

- What retry cadence, timeout behavior, and maximum delivery window should be expected from the real CRM sender?
- How long should `processed_events` be retained before archival or purging?
- Should tenant onboarding remain environment-driven, or should production use an admin-managed tenant configuration store?

## Trade-offs And Constraints

- Offset pagination is required by the assignment; cursor pagination would be better for very large datasets.
- Client configuration is environment-variable based for assignment clarity.
- The sample includes `client_a`, `client_b`, and `client_c`; additional tenants would require additional token/secret configuration or a tenant config store.
- Metrics, tracing, rate limiting, and alerting are documented as production hardening items but intentionally kept outside the assignment scope.

## Future Improvements

- Prometheus/OpenTelemetry metrics for webhook dispositions, latency, and database errors.
- Rate limiting per tenant.
- Audit log table for security-relevant events.
- Schema versioning for webhook payload evolution.
- Dead-letter handling for malformed third-party events.
- Managed secret rotation and tenant onboarding workflow.
- Cursor pagination for large contact datasets.

## Production Hardening

For a real deployment, use managed PostgreSQL, TLS, managed secrets, structured log shipping, alerting on `signature_invalid` spikes, migration approval gates, backup and restore drills, and connection-pool sizing based on workload tests.
